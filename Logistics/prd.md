# 产品需求文档 (PRD)：跨境物流成本计算器

---

## 1. Executive Summary

### Problem Statement
TikTok Shop 卖家在跨境电商业务中，需要针对不同国家和地区快速计算商品的物流成本。目前物流规则分散在多个表格中（见 `rule.md`），涵盖泰国、越南、马来西亚、菲律宾、新加坡等 5 个国家、共 12 个地区分区，每个地区有独立的起重量、起重价格和续重价格。卖家需要手动查表并按分段计费公式计算，效率低下且容易出错。

### Proposed Solution
构建一个轻量化的跨境物流成本计算器，用户输入商品重量（以克为单位），系统自动根据 `rule.md` 中的定价规则，输出所有国家和所有地区分区的物流成本明细，包括跨境物流成本、买家支付运费以及卖家最终需支付的总运费。

### Success Criteria

| 指标 | 目标值 | 测量方式 |
|------|--------|----------|
| 计算准确率 | 100% 与规则表一致 | 以 50 组已知答案的测试用例进行回归测试 |
| 响应时间 | ≤ 500ms 返回所有 12 个地区的计算结果 | 前端单次输入的端到端耗时 |
| 用户满意度 | ≥ 4.0 / 5.0 | 50 份卖家问卷 |
| 规则表配置化 | 规则表独立于代码，无需部署即可更新 | 通过加载外部文件或配置中心 |
| 零计算错误 | 上线后 30 天内无因计算错误导致的投诉 | 运营反馈数据 |

## 2. User Experience & Functionality

### User Personas

**Persona 1：TikTok Shop 跨境卖家（主要用户）**
- **角色**：个体卖家或小型电商团队运营
- **痛点**：不清楚不同国家和地区的物流费用差异，无法快速在选品时预估成本
- **期望**：输入重量后一键看到所有地区的费用，方便决策和定价

**Persona 2：运营专员（次要用户）**
- **角色**：负责定价策略和商品推广
- **痛点**：需要对比不同重量档位在各地区的成本差异
- **期望**：能够批量计算、导出结果、快速对比

**Persona 3：财务人员（次要用户）**
- **角色**：核算成本和利润
- **痛点**：需要准确的物流数据用于对账
- **期望**：计算结果可导出为表格格式

### User Stories

| ID | 用户故事 | 优先级 |
|----|---------|--------|
| US-01 | 作为一个卖家，我希望输入商品重量（克），以便系统自动计算所有国家/地区的物流成本 | P0 |
| US-02 | 作为一个卖家，我希望看到每个地区的跨境物流成本、买家支付运费和卖家总支付运费的明细，以便了解成本构成 | P0 |
| US-03 | 作为一个卖家，我希望结果按照币种区分显示（THB / VND / MYR / PHP / SGD），以便直接用于财务核算 | P0 |
| US-04 | 作为一个卖家，我希望系统在重量小于等于起重重量时，仅收取起重价格，以便符合规则表的计费逻辑 | P0 |
| US-05 | 作为一个卖家，我希望重量超过起重重量时，按每 10g 向上取整计费，以便与官方定价一致 | P0 |
| US-06 | 作为一个运营，我希望可以将计算结果导出为 CSV/Excel，以便做进一步的数据分析 | P1 |
| US-07 | 作为一个运营，我希望可以调整规则表中的价格参数，以便规则变更时无需重新部署 | P1 |
| US-08 | 作为一个用户，我希望在输入重量时能看到实时校验提示（如负值、非数字等无效输入），以便及时修正 | P1 |

### Acceptance Criteria

#### US-01 & US-02 & US-03：重量输入与全地区计算
- **Given** 用户打开物流成本计算器页面
- **When** 用户输入一个正整数重量（克）并点击「计算」
- **Then** 页面展示以下格式的表格结果，包含 5 个国家的全部 12 个地区分区：
  - 国家 / 地区 / 渠道 / 币种
  - 跨境物流成本（根据公式计算）
  - 买家支付运费（根据公式计算）
  - 卖家总支付运费 = 跨境物流成本 + 买家支付运费

#### US-04：重量 ≤ 起重重量时的计费
- **Given** 规则表中某地区跨境起重量为 10g，买家起重量为 10g
- **When** 用户输入重量 ≤ 10g（如 5g、8g、10g）
- **Then** 跨境物流成本 = 起重价格，买家支付运费 = 买家起重价格，总运费 = 两者之和

#### US-05：重量 > 起重重量时的续重计费
- **Given** 规则表中某地区跨境起重量为 10g，续重为每 10g 一个单位
- **When** 用户输入重量 > 10g（如 15g、23g、100g）
- **Then** 续重计费部分 = ⌈(重量 − 起重重量) / 10⌉ × 续重价格，向下取整到 0 的负数情况须防止出现（即取 max(0, 续重部分)）

#### US-08：输入校验
- **Given** 用户在输入框输入内容
- **When** 输入为负值、零、非数字或空值
- **Then** 展示红色提示文字「请输入大于 0 的有效重量（克）」，不触发计算

#### Non-Goals（本次不做）
- 不支持多件商品合并重量计算（v2 版本考虑）
- 不支持按目的地地址自动匹配地区分区（需要人工选择或默认全量显示）
- 不支持历史汇率换算为人民币统一展示（各币种独立显示）
- 不支持用户账户系统和登录
- 不支持移动端 App，仅限 Web 端

### 计费公式

```
续重克数 = max(0, 重量 − 起重量)
续重单位数 = ceil(续重克数 / 10)      // 每 10g 为一个续重单位，向上取整
费用 = 起重价格 + 续重单位数 × 续重价格

跨境物流成本 = 按跨境参数计算
买家支付运费 = 按买家参数计算
卖家总支付运费 = 跨境物流成本 + 买家支付运费
```

### Core Algorithm

```typescript
interface PricingRule {
  country: string;           // 国家
  region: string;            // 地区/分区
  channel: string;           // 渠道
  currency: string;          // 币种
  crossBorder: {             // 跨境物流成本
    baseWeight: number;      // 起重重量 (g)
    basePrice: number;       // 起重价格
    continuationPrice: number; // 续重价格 (每10g)
  };
  buyer: {                   // 买家支付运费
    baseWeight: number;
    basePrice: number;
    continuationPrice: number;
  };
}

interface CalculationResult {
  country: string;
  region: string;
  channel: string;
  currency: string;
  crossBorderCost: number;
  buyerShippingFee: number;
  totalSellerFee: number;
}

function calculateShippingCost(weightInGrams: number, rule: PricingRule): CalculationResult {
  // 跨境物流成本
  const cbExtraGrams = Math.max(0, weightInGrams - rule.crossBorder.baseWeight);
  const cbUnits = Math.ceil(cbExtraGrams / 10);
  const crossBorderCost = rule.crossBorder.basePrice + cbUnits * rule.crossBorder.continuationPrice;

  // 买家支付运费
  const buyerExtraGrams = Math.max(0, weightInGrams - rule.buyer.baseWeight);
  const buyerUnits = Math.ceil(buyerExtraGrams / 10);
  const buyerShippingFee = rule.buyer.basePrice + buyerUnits * rule.buyer.continuationPrice;

  // 卖家总支付
  const totalSellerFee = crossBorderCost + buyerShippingFee;

  return {
    country: rule.country,
    region: rule.region,
    channel: rule.channel,
    currency: rule.currency,
    crossBorderCost: parseFloat(crossBorderCost.toFixed(2)),
    buyerShippingFee: parseFloat(buyerShippingFee.toFixed(2)),
    totalSellerFee: parseFloat(totalSellerFee.toFixed(2)),
  };
}

function calculateAllRegions(weightInGrams: number, rules: PricingRule[]): CalculationResult[] {
  return rules.map(rule => calculateShippingCost(weightInGrams, rule));
}
```

### Integration Points

| 集成项 | 说明 |
|--------|------|
| 规则表来源 | `rule.md` → 编译为 `pricing-rules.json` |
| 币种格式化 | `Intl.NumberFormat` 按当地格式显示 |
| 导出功能 | 使用 CSV / Excel 导出库（如 `SheetJS`） |

### Security & Privacy

- **数据脱敏**：无需用户输入任何个人信息，仅输入重量数字
- **输入安全**：前端 + 后端（若有）双重校验，防止 XSS 注入
- **规则文件权限**：规则配置文件设为只读，防止意外修改
- **无敏感数据**：系统不处理任何用户隐私信息或支付数据

## 6. Risks & Roadmap

### Technical Risks

| 风险 | 影响 | 概率 | 缓解措施 |
|------|------|------|---------|
| 币种格式化在部分地区显示不一致 | 中 | 中 | 使用标准 `Intl.NumberFormat` API，针对 5 种币种编写独立测试 |
| 规则表更新时与代码逻辑脱节 | 高 | 低 | 规则表与计算函数解耦，规则变更通过配置文件管理，发布前有版本校验 |
| 浮点数精度导致计费误差 | 高 | 中 | 使用 `toFixed(2)` 统一精度，关键路径使用整数分（cents）计算 |
| `ceil()` 在不同语言/平台的行为差异 | 中 | 低 | 使用标准数学库，在单元测试中覆盖 |
